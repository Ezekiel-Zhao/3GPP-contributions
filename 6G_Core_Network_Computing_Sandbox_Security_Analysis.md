## 1. 背景与概述

### 1.1 6G通算融合趋势

6G网络的核心愿景之一是实现**通信与计算的深度融合（通算融合）**。不同于传统网络中通信与计算资源分离部署的模式，6G核心网将原生集成智算能力，在网络侧提供AI推理、数据处理、边缘智能等计算服务。

根据3GPP Release 20的6G研究项目（FS_6G_REQ, TR 22.870），6G系统将涵盖以下关键领域：

- **AI与机器学习**：网络原生AI能力
- **系统集成与运营**：整体性能优化
- **泛在连接**：多域协同
- **沉浸式通信**：高带宽低时延
- **大规模通信**：海量连接管理
- **工业与垂直行业支持**：定制化网络服务

### 1.2 智算沙箱的定义

**智算沙箱（Computing Sandbox）** 是指在6G核心网内部署的、用于运行第三方或运营商自有AI/ML计算任务的**安全隔离执行环境**。其核心特征包括：

- 与核心网信令面/用户面的**逻辑或物理隔离**
- 受控的资源访问边界（CPU、内存、存储、网络）
- 可审计、可监控的计算执行过程
- 生命周期管理（创建、运行、销毁）的自动化

### 1.3 客户安全关切的核心问题

| 关切领域 | 具体问题 |
|---------|---------|
| 数据泄露 | 智算任务是否会访问/泄露核心网用户数据？ |
| 网络攻击面 | 沙箱是否成为攻击核心网的跳板？ |
| 资源争抢 | 计算任务是否影响核心网信令的稳定性？ |
| 供应链安全 | 第三方计算代码是否引入后门？ |
| 合规性 | 是否满足电信级安全合规要求？ |

### 2.2 沙箱与核心网的交互模型

智算沙箱与核心网之间的交互遵循**最小权限原则**，通过以下机制实现：

1. **API网关隔离**：沙箱仅通过受控的API网关与核心网交互，无法直接访问核心网网元内部接口
2. **数据脱敏管道**：传入沙箱的数据经过脱敏/匿名化处理
3. **单向数据流**：默认配置为数据单向流入沙箱，结果经审计后流出
4. **零信任网络**：沙箱与核心网之间的所有通信均需认证和加密

## 3. 安全威胁分析

### 3.1 威胁模型（基于STRIDE框架）

| 威胁类别 | 威胁描述 | 风险等级 | 缓解措施 |
|---------|---------|---------|---------|
| **伪装 (Spoofing)** | 恶意沙箱伪装为合法计算节点 | 高 | 硬件级身份认证、远程证明 |
| **篡改 (Tampering)** | 沙箱内代码被篡改或注入恶意代码 | 高 | 代码签名验证、运行时完整性校验 |
| **抵赖 (Repudiation)** | 计算结果无法追溯来源 | 中 | 全链路审计日志、区块链存证 |
| **信息泄露 (Information Disclosure)** | 沙箱侧信道攻击泄露核心网数据 | 高 | TEE隔离、内存加密、噪声注入 |
| **拒绝服务 (DoS)** | 计算任务耗尽核心网资源 | 高 | 资源配额、限流、优先级调度 |
| **权限提升 (Elevation of Privilege)** | 沙箱突破隔离边界访问核心网 | 极高 | 多层隔离、硬件强制访问控制 |

### 3.2 攻击面分析

**核心攻击向量**：
1. **沙箱逃逸**：通过内核漏洞或虚拟化漏洞突破沙箱边界
2. **侧信道攻击**：通过缓存、时序、功耗等侧信道获取核心网敏感信息
3. **供应链攻击**：通过恶意第三方AI模型或库植入后门
4. **资源耗尽攻击**：通过恶意计算任务消耗共享资源
5. **API滥用**：通过合法API接口进行非预期的数据访问

## 4. 多层安全保障体系

### 4.1 第一层：硬件安全基础

| 安全机制 | 技术实现 | 保障目标 |
|---------|---------|---------|
| **硬件信任根 (HRoT)** | TPM 2.0 / Intel TXT / ARM TrustZone | 确保平台启动完整性 |
| **安全启动链** | UEFI Secure Boot + 度量启动 | 防止固件/引导层被篡改 |
| **内存加密** | Intel TME / AMD SME/SEV | 防止物理内存读取攻击 |
| **I/O隔离** | IOMMU/VT-d | 防止DMA攻击 |
| **硬件随机数** | RDRAND / TRNG | 密码学安全的随机数生成 |

### 4.2 第二层：虚拟化与隔离安全

#### 4.2.1 轻量级虚拟化隔离

- **microVM技术**（如Firecracker、Cloud Hypervisor）：
  - 每个沙箱运行在独立的microVM中
  - 启动时间 < 125ms，内存开销 < 5MB
  - 攻击面极小（仅约50K行Rust代码 vs QEMU的数百万行C代码）
  - 硬件级隔离（基于KVM）

- **安全容器**（如Kata Containers、gVisor）：
  - 容器运行在轻量级VM内
  - 内核隔离，每个容器拥有独立内核
  - 系统调用过滤和拦截

#### 4.2.2 网络微分段

```
+------------------+     +------------------+     +------------------+
|   核心网信令网    |     |   沙箱计算网络    |     |   管理控制网络    |
|   (Control Plane) |     |  (Compute Plane)  |     |  (Management)    |
+------------------+     +------------------+     +------------------+
        |                        |                        |
        +--------+-------+-------+--------+-------+-------+
                 |       |                |       |
            +----+--+ +--+---+      +----+--+ +--+---+
            | FW/ACL| | FW/ACL|      | FW/ACL| | FW/ACL|
            +-------+ +------+      +-------+ +------+
```

- **网络策略隔离**：每个沙箱拥有独立的网络命名空间和防火墙规则
- **东西向流量控制**：沙箱之间禁止直接通信
- **南北向流量审计**：所有进出沙箱的流量经过深度包检测
- **加密传输**：沙箱间通信强制使用mTLS

### 4.3 第三层：运行时安全

#### 4.3.1 可信执行环境 (TEE)

| TEE技术 | 隔离粒度 | 适用场景 |
|---------|---------|---------|
| **Intel SGX** | 进程级（Enclave） | 敏感AI推理、密钥管理 |
| **Intel TDX** | VM级（Trust Domain） | 整VM隔离的计算任务 |
| **AMD SEV-SNP** | VM级 | 大规模计算隔离 |
| **ARM CCA/RME** | Realm级 | 边缘计算场景 |
| **NVIDIA H100 CC** | GPU级 | GPU加速的AI训练/推理 |

**TEE提供的安全保障**：
- **机密性**：即使hypervisor/OS被攻破，TEE内数据仍然加密
- **完整性**：TEE内代码和数据不可被外部篡改
- **远程证明**：可向远程方证明TEE内运行的是预期代码
- **密封存储**：数据仅在特定TEE环境内可解密

#### 4.3.2 运行时安全监控

- **eBPF安全探针**：在内核层无侵入式监控沙箱行为
- **系统调用审计**：记录和审计沙箱的所有系统调用
- **异常行为检测**：基于ML的异常检测，识别沙箱逃逸尝试
- **完整性度量**：持续度量沙箱运行时完整性

### 4.4 第四层：数据安全

#### 4.4.1 数据分级保护

| 数据级别 | 数据类型 | 保护策略 |
|---------|---------|---------|
| **L1 - 公开** | 网络性能指标 | 基本访问控制 |
| **L2 - 内部** | 网络配置信息 | RBAC + 传输加密 |
| **L3 - 机密** | 用户面数据 | TEE处理 + 存储加密 |
| **L4 - 绝密** | 用户身份信息 | 脱敏/匿名化 + TEE + 审计 |

#### 4.4.2 数据脱敏与匿名化

- **差分隐私**：在数据中注入可控噪声，防止个体信息泄露
- **联邦学习**：数据不出本地，仅交换模型梯度
- **同态加密**：在加密数据上直接进行计算
- **安全多方计算**：多方联合计算，各方数据互不可见

### 4.5 第五层：编排与管理安全

#### 4.5.1 沙箱生命周期安全

```
[请求提交] --> [身份认证] --> [代码审查] --> [资源配额分配]
     |              |              |              |
     v              v              v              v
[沙箱创建] --> [安全启动验证] --> [运行时监控] --> [结果审计]
     |              |              |              |
     v              v              v              v
[安全销毁] <-- [资源回收] <-- [日志归档] <-- [合规检查]
```

#### 4.5.2 安全策略引擎

- **RBAC + ABAC混合授权**：基于角色和属性的细粒度访问控制
- **动态策略评估**：基于实时安全态势动态调整沙箱权限
- **合规自动化**：自动检查沙箱配置是否符合安全基线
- **变更管理**：所有安全策略变更需经过审批流程

## 5. 3GPP标准化安全框架

### 5.1 当前标准化进展

| 研究项目 | 工作组 | 内容 |
|---------|-------|------|

| **FS_6G_ARC** (Study on Architecture for 6G System) | SA WG2 | 6G系统架构研究（含安全架构） |

| **FS_CryptoPQC** (Study on Transitioning to Post Quantum Cryptography) | SA WG3 | 后量子密码迁移研究 |
| **FS_6G_LI** (Study on Lawful Interception for 6G) | SA WG3LI | 6G合法拦截研究 |

### 5.2 5G安全基础的继承与增强

6G安全体系将建立在5G安全架构（3GPP TS 33.501）的坚实基础之上，并进一步增强：

**继承的5G安全能力**：
- **SEPP（安全边缘保护代理）**：保护运营商间信令安全
- **SUCI/SUPI保护**：用户永久标识加密保护
- **NAS/AS安全**：非接入层和接入层的双层安全
- **网络切片隔离安全**：切片间的安全隔离机制
- **服务化架构安全**：基于OAuth 2.0的网元间认证授权

**6G增强方向**：
- **AI/ML安全**：AI模型和训练数据的安全保护
- **计算安全**：网络内计算任务的安全隔离与审计
- **后量子密码**：抗量子计算攻击的密码算法迁移
- **零信任架构**：从边界安全向零信任安全模型演进
- **分布式信任**：去中心化的信任管理

### 5.3 Release 21安全规范展望

Release 21作为首个6G规范性版本（预计2027年3月Stage-1冻结，2028年12月Stage-3冻结），将制定6G安全的强制性技术规范，预计包含：

- 6G系统安全架构规范
- 计算功能安全要求
- 增强型认证与密钥管理
- AI/ML功能安全框架
- 后量子密码集成方案

## 6. 关键安全隔离技术

### 6.1 技术对比矩阵

| 隔离技术 | 隔离强度 | 性能开销 | 启动速度 | 适用场景 |
|---------|---------|---------|---------|---------|
| **物理隔离** | 极高 | 高 | 慢 | 最高安全等级场景 |
| **硬件TEE** | 高 | 低-中 | 快 | 敏感计算、密钥管理 |
| **microVM** | 高 | 低 | 极快 | 通用计算沙箱 |
| **安全容器** | 中-高 | 低 | 快 | 轻量级应用隔离 |
| **标准容器** | 中 | 极低 | 极快 | 可信内部应用 |
| **进程隔离** | 低 | 极低 | 极快 | 低风险内部工具 |

### 6.2 推荐的多层隔离方案

针对6G核心网内智算沙箱，推荐采用**纵深防御（Defense in Depth）**策略：

```
+----------------------------------------------------------+
| 最外层: 网络微分段 (Network Micro-segmentation)           |
|   - 独立VLAN/VXLAN                                       |
|   - 防火墙策略                                            |
|   - 流量审计                                              |
+----------------------------------------------------------+
    |
    v
+----------------------------------------------------------+
| 第二层: 虚拟化隔离 (Virtualization Isolation)             |
|   - microVM / 安全容器                                    |
|   - 独立内核空间                                          |
|   - 资源配额限制                                          |
+----------------------------------------------------------+
    |
    v
+----------------------------------------------------------+
| 第三层: 可信执行环境 (TEE)                                |
|   - Intel TDX / AMD SEV-SNP / ARM CCA                    |
|   - 内存加密                                              |
|   - 远程证明                                              |
+----------------------------------------------------------+
    |
    v
+----------------------------------------------------------+
| 最内层: 应用级安全 (Application Security)                  |
|   - 代码签名验证                                          |
|   - 运行时完整性度量                                      |
|   - 最小权限执行                                          |
+----------------------------------------------------------+
```

### 6.3 关键隔离技术详解

#### 6.3.1 Intel TDX (Trust Domain Extensions)

- **隔离粒度**：整个VM（Trust Domain）
- **保护范围**：VM内存、VM寄存器、VM控制结构
- **威胁模型**：即使VMM/hypervisor被攻破，TD内数据仍受保护
- **远程证明**：支持TD级别的远程证明，可验证TD内运行的软件
- **适用场景**：运行完整OS的智算任务

#### 6.3.2 Firecracker microVM

- **安全设计**：Rust编写，内存安全，无legacy设备支持
- **攻击面**：约50K行代码（vs QEMU数百万行）
- **隔离保证**：基于KVM的硬件级隔离
- **资源限制**：内置CPU、内存、网络I/O限流
- **适用场景**：无服务器计算、边缘智算沙箱

#### 6.3.3 ARM CCA (Confidential Compute Architecture)

- **Realm管理**：通过RMM（Realm Management Monitor）管理隔离域
- **粒度灵活**：支持从进程到VM的多粒度隔离
- **边缘适用**：适合6G边缘计算节点的受限环境
- **生态支持**：ARMv9架构原生支持

## 7. 业界安全最佳实践与标准

### 7.1 相关标准与框架

| 标准/框架 | 发布组织 | 与6G智算沙箱的关联 |
|----------|---------|------------------|
| **NIST SP 800-207** | NIST | 零信任架构指南 |
| **NIST SP 800-190** | NIST | 容器安全指南 |
| **NIST SP 800-125B** | NIST | 安全虚拟化指南 |
| **ETSI TS 133 501** | ETSI/3GPP | 5G系统安全规范 |
| **GSMA FS.40** | GSMA | 5G安全指南 |
| **CIS Benchmarks** | CIS | 容器/VM安全基线 |
| **Confidential Computing Consortium** | CCC | 机密计算标准 |

### 7.2 零信任安全模型

6G智算沙箱的安全设计遵循零信任原则：

1. **永不信任，始终验证**：沙箱默认被视为不可信
2. **最小权限**：沙箱仅获得完成任务所需的最小权限
3. **假设被攻破**：安全设计假设沙箱可能已被攻破
4. **持续验证**：运行时持续验证沙箱的安全状态
5. **微分段**：网络层面的细粒度隔离

### 7.3 机密计算联盟 (CCC) 标准

机密计算联盟定义的安全属性直接适用于6G智算沙箱：

- **数据使用中的保护**：数据在计算过程中始终加密
- **代码完整性**：确保执行的代码未被篡改
- **可验证性**：通过远程证明验证计算环境
- **数据隔离**：不同计算任务的数据严格隔离

### 7.4 电信行业安全合规

| 合规要求 | 覆盖范围 | 智算沙箱的合规措施 |
|---------|---------|------------------|
| **GDPR** | 用户数据保护 | 数据脱敏、差分隐私、联邦学习 |
| **NIS2指令** | 关键基础设施安全 | 多层隔离、安全监控、事件响应 |
| **3GPP SCAS** | 网络安全保障 | 遵循3GPP安全测试规范 |
| **Common Criteria** | IT安全评估 | 关键组件通过EAL4+认证 |
| **ISO 27001** | 信息安全管理 | 安全管理体系认证 |

## 8. 客户常见安全挑战与应答

### 8.1 挑战1："计算任务运行在核心网内，一旦被攻破，核心网就暴露了"

**应答要点**：

> 6G智算沙箱采用**纵深防御**架构，即使沙箱被攻破，攻击者仍面临多重安全屏障：
>
> 1. **硬件级隔离**：沙箱运行在TEE（如Intel TDX/AMD SEV-SNP）中，即使hypervisor被攻破，TEE内数据仍受硬件保护
> 2. **网络微分段**：沙箱与核心网信令面/用户面之间通过防火墙和ACL严格隔离，沙箱无法直接访问核心网网元
> 3. **最小权限API**：沙箱仅能通过受控API访问经脱敏处理的数据，无法获取原始用户数据
> 4. **持续监控**：运行时安全监控引擎实时检测异常行为，自动触发沙箱隔离或销毁
>
> **类比**：这就像在银行大楼内设置了一个独立的ATM机，它有独立的保险箱（TEE）、独立的门禁系统（网络隔离）、只能看到脱敏后的账户信息（数据脱敏），并且有24小时监控（安全监控）。即使ATM机被攻破，攻击者也无法进入银行金库。

### 8.2 挑战2："第三方AI模型可能包含恶意代码"

**应答要点**：

> 针对供应链安全，我们建立了完整的安全防线：
>
> 1. **代码签名与验证**：所有AI模型必须经过数字签名，沙箱启动时验证代码完整性
> 2. **静态安全扫描**：模型部署前进行恶意代码扫描和漏洞检测
> 3. **沙箱隔离执行**：即使模型包含恶意代码，也被限制在沙箱内，无法影响核心网
> 4. **行为监控**：运行时监控模型的实际行为，检测异常模式
> 5. **远程证明**：通过TEE远程证明，确保沙箱内运行的是经过审核的代码

### 8.3 挑战3："计算任务会不会抢占核心网资源，导致网络故障？"

**应答要点**：

> 资源安全是设计的核心考量之一：
>
> 1. **物理资源隔离**：智算沙箱使用专用的计算资源池，与核心网信令处理资源物理分离
> 2. **严格资源配额**：每个沙箱有明确的CPU、内存、存储、网络带宽配额上限
> 3. **优先级调度**：核心网信令处理始终具有最高优先级，计算任务为低优先级
> 4. **自动限流**：当计算任务资源使用超过阈值时，自动限流或终止
> 5. **故障隔离**：计算任务故障不会影响核心网的正常运行

### 8.4 挑战4："如何保证用户数据不被泄露？"

**应答要点**：

> 用户数据保护采用多层防护策略：
>
> 1. **数据分级**：用户数据按敏感程度分级，不同级别有不同的处理策略
> 2. **脱敏处理**：传入沙箱的数据经过脱敏/匿名化处理，去除可识别信息
> 3. **TEE保护**：敏感数据仅在TEE内处理，即使OS/hypervisor被攻破也无法读取
> 4. **联邦学习**：支持数据不出本地的分布式学习模式
> 5. **差分隐私**：在计算结果中注入噪声，防止通过结果反推个体信息
> 6. **审计追溯**：所有数据访问和处理操作均有完整审计日志

### 8.5 挑战5："6G标准还没出来，安全性如何保证？"

**应答要点**：

> 6G安全标准正在快速推进中，但安全能力并非从零开始：
>
> 1. **继承5G安全基础**：6G安全架构建立在经过验证的5G安全体系（3GPP TS 33.501）之上
> 2. **3GPP Rel-20安全研究**：SA WG3已启动"FS_6G_SEC"研究项目，系统性研究6G安全架构
> 3. **后量子密码准备**：同步启动"FS_CryptoPQC"研究，为后量子密码迁移做准备
> 4. **业界成熟技术**：TEE、microVM、零信任等安全隔离技术已在云计算、金融等行业大规模验证
> 5. **标准时间线明确**：Rel-21（首个6G规范版本）预计2027年Stage-1冻结，2028年完成全部规范

## 9. 结论与建议

### 9.1 核心结论

1. **安全可行性已充分论证**：通过多层纵深防御架构（硬件TEE + 虚拟化隔离 + 网络微分段 + 运行时监控 + 数据安全），6G核心网内运行智算沙箱的安全性可以得到充分保障。

2. **标准化进程明确**：3GPP已在Rel-20启动6G安全研究（FS_6G_SEC），Rel-21将制定强制性安全规范，安全标准体系正在快速完善。

3. **技术成熟度高**：核心安全隔离技术（TEE、microVM、零信任）已在云计算、金融、国防等行业大规模部署验证，技术成熟度足以支撑电信级应用。

4. **继承5G安全优势**：6G安全不是从零开始，而是建立在5G安全架构（TS 33.501）经过全球运营商验证的坚实基础之上。

5. **合规体系完善**：安全设计覆盖GDPR、NIS2、3GPP SCAS、ISO 27001等主要合规要求。
